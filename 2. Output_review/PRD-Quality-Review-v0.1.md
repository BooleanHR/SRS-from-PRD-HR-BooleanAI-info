## 2. 챕터·파트별 수정 방향

### §1 개요·목표

**문제점:**
- `KPI-5 감사 리포트 자동 생성률`: 측정 창구(어느 시스템에서 어떻게 집계하는지)가 서술 없음
- 북극성 KPI 기준선이 "0% (현재 수기 캡처 의존)"으로만 표현되어 정의가 모호 (수기 = 0%인 이유, 현재 상태 정의 필요)

**수정 방향:**
- 각 KPI에 "측정 쿼리/집계 방법" 컬럼 추가
- 북극성 KPI의 분자/분모 정의를 명문화

---

### §3 사용자 스토리와 AC

**문제점:**

| AC | 문제 |
|---|---|
| Story-1 AC-1 | 실패 케이스 1개 (기관 로딩 지연)만 존재. 캡처 자체가 완전히 불가한 상황(기관 API 전면 차단) 시나리오 미포함 |
| Story-2 AC-2 | 실패 케이스: "API 전면 불가" 시 담당자 알림 도구가 SMS로만 지정되어 있어 대체 채널 부재 시 단절 위험 |
| Story-3 AC-1 | 알림톡 재시도 1회 + SMS 폴백만 명시. SMS도 실패한 경우(번호 오류 등) 처리 누락 |
| Story-3 AC-3 | Self-Service 처리율 ≥ 90%의 측정 분모 정의 불명확 (전체 불일치 건 기준인지, 알림톡 발송 건 기준인지) |
| Story-4 AC-1 | 외부 API 응답 실패 시 파트너 ATS에 어떤 폴백 응답이 가는지 미정의 |

**수정 방향:**
- 각 AC에 실패 케이스 최소 2개 이상으로 보강
- Self-Service 처리율의 분자/분모를 명확하게 재정의

---

### §4 기능 요구사항

**문제점:**
- Should/Could 항목에 "1스프린트 내 구현 가능성" 검토 부재 (스프린트 = 2주 기준 시 구현 가능한 단위인지 언급 없음)
- M2(공공 API/RPA 기관 조회)와 M3(병렬 캡처)의 의존성(정부24 API 계약 완료 전제) 명시 누락

**수정 방향:**
- 각 MSCW 항목에 예상 공수(T-shirt sizing: S/M/L)와 전제 의존성 추가

---

### §5 비기능 요구사항

**문제점:**
- **비용 임계치 전무**: NFR에 성능/신뢰성/보안은 있으나 운영 비용 관련 임계치 없음
  - 예: 공공 API 호출 비용 한도, OCR 처리 건당 단가 상한
- **RPA 감지 기준 불명확**: "기관 사이트 변경 감지" 알림의 트리거 조건이 정의되지 않음
  - 어떤 신호로 "변경"을 판단하는지 (HTTP 상태코드 변화? 응답 DOM 해시 변화?)
- 가용성 SLO(99.5%)와 RPA 복구 SLA(72시간)이 동일 문서에 있으나 상충 가능성 미검토
  - 72시간 RPA 중단 시 가용성 SLO 위반 여부에 대한 정의 없음

**수정 방향:**
- 비용 임계치 섹션 신설 (API 호출 월 한도, 초과 시 알림 기준)
- RPA 감지 기준을 DOM 해시 비교 주기와 알림 트리거 조건으로 명문화
- RPA 중단 중 가용성 SLO 예외 처리 정의

---

### §8 실험·롤아웃·측정

**문제점:**
- **Exp-1 통계 유의성 미정의**: "A그룹 점수 ≥ B그룹 + 1.0점"이 성공 기준이나, n=50으로 유의성 검정 가능 여부 및 통계 방법(t-test 등) 미명시
- **Exp-2 Before-After 설계**: 계절성/채용 규모 차이로 인한 혼란 변수 통제 방법 미언급
- **Exp-3 PoC 성공 이후 확장 기준**: PoC 성공 시 본 계약 전환 기준(예: PoC 고객의 구독 전환율 ≥ X%) 미정의

**수정 방향:**
- Exp-1: 표본 크기 산출 근거 또는 유의수준 명시 (α=0.05, 단측 t-test)
- Exp-2: 동일 기관 동일 회차 규모 통제 방법 명시
- PoC→본계약 전환율 KPI 추가

---

### §9 근거 (Proof)

**문제점:**
- 대부분의 참고 링크가 "[예정]" 처리 → 현재 ADR 링크 실체 없음
- JTBD 인터뷰 계획은 VPS에 있으나 PRD 내 인터뷰 완료 기준(언제, 누가, 어떻게 수행했는지)이 공백

**수정 방향:**
- 현재 확보된 1차 데이터(VPS 분석, 파트너사 실사 등)와 미확보 데이터를 명확히 분리
- 인터뷰 실행 완료 기준일 및 책임자 명시

### [수정] §3 Story-1 AC-1 — 실패 케이스 추가

**AC-1 | 병렬 캡처 완결성** *(수정본)*

- **Given** 지원자가 자격증/학위 서류를 시스템에 업로드하고 검증이 완료된 상태에서
- **When** 검증 엔진이 공공 API 또는 RPA를 통해 기관 사이트를 조회할 때
- **Then** 원본 서류 이미지 + 기관 조회 결과 화면이 단일 프레임 내에 나란히 캡처되어 저장되어야 한다.
  - 임계치: 캡처 성공률 **≥ 99.5%** (배치당 n ≥ 100건) / 타임스탬프 정밀도 **± 1초 이내**
  - **실패 케이스 1:** 기관 사이트 로딩 지연 (응답 > 10초) → 재시도 2회 후 '수동 확인 큐' 이관 + 담당자 대시보드 플래그 표시
  - **실패 케이스 2:** 기관 API/RPA 전면 차단 (HTTP 403 또는 연속 실패 5회) → '기관 접속 불가' 시스템 배너 즉시 표시 + 담당자·온콜 엔지니어 동시 Slack 알림 발송 + 해당 회차 검증 일시 중지 처리
  - **실패 케이스 3:** 캡처 이미지 저장 실패 (스토리지 오류) → 캡처 임시 버퍼 재시도 1회, 최종 실패 시 해당 건 Audit Trail 'CAPTURE_FAILED' 상태로 기록 + QA 수동 검토 큐 이관

### [수정] §3 Story-3 AC-1 및 AC-3 — 실패 케이스 및 분모 정의 추가

**AC-1 | 불일치 감지 후 알림톡 발송 속도** *(수정본)*

- **Given** Triple Check Loop 결과 '불일치' 또는 '서류 미비' 상태가 확정된 상태에서
- **When** 검증 엔진이 결과를 저장하면
- **Then** 해당 지원자의 등록 연락처로 카카오알림톡이 **3분 이내** 자동 발송되어야 한다.
  - 임계치: 알림톡 발송 성공률 **≥ 99%** / 발송 지연(> 3분) 건에 대한 재시도 1회
  - **실패 케이스 1:** 알림톡 재시도 실패 → SMS 폴백 자동 발송 (1분 이내)
  - **실패 케이스 2:** SMS 폴백도 실패 (번호 오류·수신 거부) → 담당자 대시보드에 '알림 실패 - 수동 연락 필요' 플래그 표시 + 해당 지원자 목록 CSV 내보내기 버튼 활성화
  - **실패 케이스 3:** 지원자 연락처 미등록 → 알림 발송 시도 없이 즉시 담당자에게 '연락처 없음' 건 표시 (알림 성공률 분모에서 제외하여 지표 왜곡 방지)

**AC-3 | 민원 전화 감소 측정 가능성** *(수정본)*

- **Given** Self-Service 루프가 활성화된 채용 회차에서
- **When** 해당 회차가 종료되면
- **Then** 시스템이 아래 지표를 회차별로 자동 집계하여 대시보드에 표시해야 한다.

  **측정 정의 명확화:**
  - **분모 (전체 처리 대상):** 알림톡 발송이 성공적으로 완료된 건수 (연락처 없음 건 제외)
  - **분자 (Self-Service 처리):** 알림톡 발송 후 72시간 이내 지원자 재제출 완료 건수 (담당자 개입 없음)
  - **수동 개입 건:** 위 두 지표에 해당하지 않아 담당자가 직접 연락하거나 큐에서 수동 처리한 건수

  **집계 항목:**
  1. 알림톡 발송 건수
  2. 72시간 이내 재제출 완료 건수 (Self-Service 처리율 분자)
  3. 담당자 수동 개입 건수
  4. 미응답(72시간 초과) 건수

  - 임계치: Self-Service 처리율 = (재제출 완료 건) / (알림톡 발송 완료 건) × 100 **≥ 90%**
  - 리포트 생성 응답 시간 **≤ 10초**
  - **실패 케이스:** Self-Service 처리율 < 80% 연속 2회 차 → 알림톡 템플릿 재검토 또는 링크 만료 기간 연장(48h → 96h) 검토 알림

### [수정] §8 실험 계획 — 통계 유의성 및 혼란 변수 통제 추가

#### Experiment-1 | 병렬 캡처 기반 면책권 체감도 검증 *(수정본)*

| 항목 | 내용 |
|---|---|
| **가설** | 병렬 캡처 리포트를 제공받은 담당자는 수기 스크린샷 방식 대비 감사 대응 자신감이 유의미하게 높다. |
| **설계** | A/B 테스트: A그룹(병렬 캡처 리포트) vs B그룹(기존 수기 스크린샷), n = 각 50명 (공공기관 채용 담당자) |
| **통계 방법** | 독립표본 단측 t-test (α = 0.05, 검정력 0.80) / 표본 크기 n=50은 Cohen's d = 0.5 기준 충족 |
| **혼란 변수 통제** | 무작위 배정(랜덤 시드 배정), 동일 직급(PM·담당자급) 한정, 동일 기관 규모(500명 이상 공채 경험자) 한정 |
| **측정 지표** | 감사 대응 자신감 점수 (Likert 5점, 3문항 평균) / 리포트 활용 빈도 (채용 회차당 다운로드 횟수) |
| **성공 기준** | A그룹 평균 ≥ B그룹 평균 + 1.0점 (p < 0.05) |
| **측정 도구** | 인앱 설문 (SurveyMonkey 또는 동등 도구) + 시스템 다운로드 이벤트 로그 |

#### Experiment-2 | Self-Service 루프 민원 감소 검증 *(수정본)*

| 항목 | 내용 |
|---|---|
| **가설** | 알림톡 Self-Service 루프 도입 시 담당자 수동 개입 건수가 90% 이상 감소한다. |
| **설계** | Before-After 비교: 도입 전 2회 채용 회차 vs 도입 후 2회 채용 회차 (동일 기관 기준) |
| **혼란 변수 통제** | ① 동일 기관, ② 유사 공채 규모 (±20% 이내), ③ 동일 시즌(계절성 배제: 상반기 vs 상반기) 조건 충족 시 비교 |
| **측정 지표** | 회차당 담당자 수동 전화 건수 / Self-Service 처리율 (%) / 재제출 완료 소요 시간 (평균, 시간 단위) |
| **성공 기준** | 수동 개입 건수 ≥ 90% 감소 / 재제출 완료율 ≥ 85% / 재제출 소요 시간 ≤ 24시간 (중위값) |
| **측정 도구** | 시스템 이벤트 로그 (notification → resubmit_complete 펀넬) + 담당자 타임트래킹 설문 (5분 내 응답, 주 2회) |
